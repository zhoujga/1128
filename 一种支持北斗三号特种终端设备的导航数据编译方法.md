# 一种支持北斗三号特种终端设备的导航数据编译方法

## 专利信息
- **申报类型**: 发明
- **发明人**: 周建国
- **技术领域**: 北斗三号导航技术领域、特种移动终端导航数据处理技术领域

---

## 背景技术

北斗三号导航系统是我国自主研发的卫星导航系统第三代，具备高精度、高可靠性等特点，广泛应用于位置服务、导航应用、军事用途、交通运输、灾难应急响应等领域。

### 北斗三号特种终端设备特点
- 短报文单次传输容量最大为14000比特（约1000个汉字）
- 通信时效成本较高
- 卫星信道资源紧张导致实时性受限
- 采用嵌入式处理器，存储空间和内存容量有限

### 现有技术问题

1. **数据不兼容**：传统高德导航、百度导航等主流导航软件基于地面移动通信网络设计，依赖云端实时数据下载
2. **无法离线导航**：存在数据格式不兼容、依赖网络实时更新机制
3. **存储结构不适应**：无法支持离线导航场景、不适应北斗三号嵌入式移动终端环境

### 技术问题
- 数据体积庞大（通常达到数GB级别）
- 查询效率低下（时间复杂度为O(n)）
- 更新困难（需整体替换）
- 跨区域路径规划复杂（缺乏网格边界拓扑关系管理）
- 技术门槛高（需专业人员设计二进制数据结构、偏移量计算、索引维护等）

---

## 发明目的

1. **解决通信带宽和终端资源受限问题**：针对北斗三号特种终端设备面临的通信带宽有限、通信时效成本较高、终端资源受限等问题
2. **实现高效存储与检索**：通过网格化数据组织与SQLite轻量级存储，实现导航数据的高效存储、快速检索与增量更新
3. **降低技术门槛**：通过采用SQLite数据库替代传统二进制数据块加文件索引的编译方式，使中小型团队也能独立完成导航数据的编译与维护

---

## 技术方案

### 总体流程

```
数据采集与整合 → 数据清洗与拓扑修复 → 网格归属计算与动态分裂 
    → 拓扑关系构建 → 数据格式化与SQLite入库 → 索引创建与性能优化 
    → 增量更新与维护
```

---

### （一）数据采集和整合

**数据来源**：
- 北斗三号原始定位数据
- 基础测绘成果（1:5万、1:25万矢量地形图数据）
- GIS矢量数据（Shapefile格式）
- POI兴趣点文本数据
- 权威部门发布的道路网数据

**处理步骤**：
1. **格式转换**：将SHP、KML、GeoJSON等不同来源的GIS数据转换为统一中间格式
2. **坐标统一**：所有空间数据坐标系统一转换至WGS84坐标系
3. **数据关联**：将同一条道路的不同来源数据进行匹配与融合
4. **属性标准化**：统一道路等级、名称、宽度等属性的编码标准

---

### （二）数据清洗与拓扑修复

**第一步：数据清洗**
- 剔除重复数据（基于道路中心线坐标和名称进行重复检测与合并）
- 剔除错误数据（过滤坐标越界、属性异常的记录）
- 异常值处理（修正明显错误的坐标点、长度异常的道路）

**第二步：拓扑修复**
- 断链修复（检测并修复道路弧段之间的断裂，使其首尾相连）
- 孤立结点处理（删除或关联无法构成通路的孤立结点）
- 环岛处理（识别环岛路口，创建复合结点表示多向连接）
- 假结点清除（合并连续且无分支的冗余结点）
- 边界面处理（确保跨网格边界道路的连续性）

---

### （三）地理编码与逆地理编码处理

**地理编码（地址转坐标）**：
1. 地址分词：对待编码地址进行分词处理，提取省市区、道路名、门牌号等信息
2. 道路匹配：根据道路名称在道路表中进行匹配，获取候选道路
3. 位置精确定位：结合门牌号范围，计算最佳匹配坐标

**逆地理编码（坐标转地址）**：
1. 最近道路搜索：在道路表中搜索距离目标点最近的弧段
2. 匹配容差：设定合理距离阈值，如50米，超过阈值返回失败
3. 插值计算：基于最近弧段的形状点进行线性插值，获取更精确的位置
4. POI关联：搜索周边POI，返回最近的地名或兴趣点信息

---

### （四）网格归属计算与动态分裂

#### 第一步：网格划分 - 两级网格体系

**一次网格**：
- 覆盖较大区域
- 以经差1°、纬差40′为基本单元
- 经差1°对应纬度方向约111km，满足区域级导航的覆盖需求
- 纬差40′对应经度方向约50-60km，单个网格道路数据量适中

**二次网格**：
- 在一次网格基础上细分为8×8格
- 每格大小为经差7.5′、纬差5′
- 对应实地约10×10公里，与1:2.5万地形图分幅一致

#### 第二步：网格编号规则

**一次网格编号（4位整数RRLL）**：
- RR为纬度行号，LL为经度列号
- 起点为纬度0°、经度60°

**二次网格编号（RRLLMN）**：
- 在一次网格编号基础上尾加两位数字
- M为纬度行号（0-7），N为经度行号（0-7）

**计算公式**：
```
RR = INT(纬度 × 60 / 40)
LL = INT(经度) - 60
M = INT((纬度 - RR × 40/60) × 60 / 5)
N = INT((经度 - LL - 60) × 600 / 75)
```

**示例**：经度116.4375°、纬度39.875° → 一次网格号5956，二次网格号595663

#### 第三步：动态网格选择规则

| 条件 | 处理方式 |
|------|----------|
| 道路数量 < 2048 | 采用一次网格存储 |
| 道路数量 ≥ 2048 | 自动触发二次网格划分 |

**阈值确定依据**：
- 终端设备内存限制：可用内存通常为2GB-8GB，单个网格数据加载需控制在256-512MB以内
- 查询性能：SQLite单表数据量超过2000条后查询性能开始明显下降
- 统计数据支撑：有效一次网格数量为1425个，有效二次网格数量为55469个

#### 第四步：正规化坐标表示

**计算公式**：
```
Coord_norm = (Coord_abs - Coord_ref) / Coord_span
```
- Coord_abs：绝对坐标
- Coord_ref：网格参考点（左下角点）的绝对坐标
- Coord_span：网格的坐标跨度

**精度**：
- 一次网格：经度1.52587890625e-5°、纬度1.017252604167e-6°
- 二次网格：经度1.9073486328125e-6°、纬度1.27156575520833e-6°

---

### （五）拓扑关系构建

#### 第一步：网格内拓扑
- 结点关联弧段：遍历弧段表，识别每条弧段的起止结点，填充node表的link_ids字段
- 转向关系提取：分析同一结点的出弧段集合，构建转向连接关系
- 禁行规则录入：录入单行道、限行时间等交通限制规则

#### 第二步：跨网格拓扑
- 边界点识别：识别位于网格边界的结点，标记boundary_flag=1
- 邻接关系记录：在conn_region字段记录相邻网格编号，0-7方向
- 跨网格弧段拆分：将跨越网格边界的弧段拆分为两段，分别存储在对应网格中

---

### （六）数据格式化与SQLite入库

#### 第一步：数据表结构设计

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| grid_info | 网格元数据 | grid_id, level, min_lon, min_lat, max_lon, max_lat, road_count, node_count, link_count |
| node | 结点数据 | node_id, grid_id, lon, lat, link_ids, boundary_flag, topo_layer, conn_region |
| link | 弧段数据 | link_id, grid_id, road_id, start_node, end_node, length, direction, geometry(BLOB), max_speed, road_class, func_class, toll_flag |
| road | 道路数据 | road_id, name, name_id, link_list, road_class, width, speed_class, toll_flag |
| poi | 兴趣点数据 | poi_id, grid_id, lon, lat, name, type, category |
| topology | 拓扑关系 | from_link, to_link, turn_cost, turn_restrict |

#### 第二步：几何数据压缩存储

**压缩方法**：差分编码 + 行程长度编码（RLE）

**压缩流程**：
1. 首点存储绝对坐标（经纬度）
2. 后续点存储与前一点的坐标差值
3. 对重复的差值使用行程长度编码
4. 最终数据转换为二进制BLOB存储

**压缩比**：3:1 ~ 5:1

#### 第三步：数据格式化
1. 坐标正规化：将绝对坐标转换为相对网格参考点的正规化坐标
2. 几何压缩：对弧段geometry字段进行差分编码+RLE压缩
3. 数据编码：将道路等级、方向等属性按设计编码为整数
4. 文件拆分：按网格编号拆分数据，生成独立的SQLite数据库文件（如5956.db、595663.db）

#### 第四步：索引优化策略

| 索引名 | 用途 |
|--------|------|
| idx_grid_level | 按网格级别快速筛选 |
| idx_grid_valid | 按道路数量排序查询 |
| idx_node_grid | 按所属网格定位结点 |
| idx_node_coord | 按坐标范围查询邻近结点 |
| idx_link_grid | 按所属网格定位弧段 |
| idx_link_nodes | 按起止结点查找拓扑关系 |
| idx_poi_grid | 按网格筛选POI |
| idx_poi_type | 按类型分类查询 |

**数据库优化操作**：
- 运行ANALYZE语句更新表统计信息
- 运行VACUUM命令整理数据库文件
- 设置PRAGMA参数（synchronous=NORMAL、cache_size等）
- 对最终的.db文件进行压缩

---

### （七）增量更新与维护机制

#### 第一步：增量更新策略

| 要素 | 说明 |
|------|------|
| 更新单元 | 以网格为最小更新单元 |
| 版本管理 | 为每个网格数据维护版本号，支持回滚 |
| 更新包结构 | 包含网格编号、更新类型、变化数据、校验码 |

#### 第二步：更新传输方式

| 方式 | 适用场景 |
|------|----------|
| 北斗短报文 | 小数据量更新（几个网格的POI变化、属性变更） |
| 北斗定位终端数据口 | 中等数据量更新（通过USB/RS232串口同步数据） |
| 外部存储介质 | 大数据量完整更新（道路拓扑重大变化，通过SD卡等） |

#### 第三步：更新流程

```
接收更新包 → 备份当前数据 → 执行更新 → 验证一致性 → 回滚处理
```

## 有益效果

### 与现有技术对比

| 指标 | 现有技术 | 本发明 |
|------|----------|--------|
| 数据体积 | 数GB级别 | 减少40%~60% |
| 查询效率 | O(n) | O(1) |
| 更新方式 | 整体替换 | 增量更新，减少70%+ |
| 技术门槛 | 高（需专业人员） | 低（SQLite+SQL） |

### 具体效果

1. **两级网格体系与动态分裂规则**：道路数量≥2048自动触发二次网格划分，实现数据分块存储与负载均衡
2. **正规化坐标与混合压缩**：存储空间减少约40%~60%，单个SQLite数据库文件仅需数百KB即可启动
3. **网格编号直接定位**：实现O(1)时间复杂度空间查询，配合空间索引进一步加速检索
4. **网格级增量更新**：数据更新量减少70%以上，支持多种传输方式
5. **网格边界拓扑关系**：有效解决跨区域路径规划复杂的问题
6. **SQLite替代二进制**：大幅降低导航数据编译工作的技术门槛

---

## 总结

本发明实现了导航数据的高效存储、快速检索与增量更新，满足北斗三号特种终端设备对离线导航数据的严苛要求：

- ✅ 高效存储（压缩比3:1~5:1）
- ✅ 快速检索（O(1)复杂度）
- ✅ 增量更新（减少70%+更新量）
- ✅ 低带宽适应（支持北斗短报文）
- ✅ 低技术门槛（SQLite+SQL）
