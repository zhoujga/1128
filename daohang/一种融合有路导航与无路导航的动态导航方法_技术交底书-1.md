## 所属技术领域

本发明属于导航定位技术领域，具体涉及一种融合有路导航引擎与无路导航引擎的动态导航方法，尤其适用于全地形越野导航、航路导航等复杂场景下的实时导航应用。

## 背景技术

随着越野驾驶、户外探险、军事作战、无人机航路飞行等应用场景的需求日益增长，导航系统面临着越来越复杂的应用环境。从技术实现角度，导航系统可分为两大主流技术路线：

第一类技术路线为有路导航（有网导航）技术，以高德导航、百度导航、腾讯导航等互联网地图服务商提供的导航服务为代表。该类技术基于预先构建的道路网络数据进行路径匹配和导航引导，通过GPS定位点与道路网络的匹配计算当前位置，并根据已规划路径提供转弯提示、距离播报等导航服务。该技术在城市道路、高速公路等道路网络完善的区域具有较高的定位精度和导航准确性，其核心技术包括道路网络拓扑构建、地图匹配算法、路径规划算法（如A*算法、Dijkstra算法）、转弯提示生成算法等。然而，该类技术依赖于完整且现势性好的道路网数据，当用户进入没有道路网络覆盖的区域时，定位匹配失败率显著上升，导航服务完全中断。

第二类技术路线为无路导航（无网导航）技术，以Garmin手持GPS设备、大疆无人机飞控系统、军事战术导航系统等为代表。该类技术在没有道路网络的区域（如荒野、山地、水域、空中）进行导航，根据预设航路点或目标点，通过航向角计算、距离计算等方式引导用户到达目的地。典型应用包括户外探险中的轨迹导航、无人机飞行中的航线规划、军事作战中的越野机动等。该类技术的核心算法包括航路点管理、航向角计算、距离计算、偏航检测等，但缺乏对道路网络的匹配能力，无法利用已有的道路信息提升导航精度。

现有技术在实际应用中主要存在以下技术问题：

第一，单引擎导航的适用性局限问题。现有导航系统通常只采用单一导航引擎架构，无法适应复杂多变的实际环境。以有路导航为例，高德导航SDK、百度导航SDK等主流产品均基于单一的道路匹配引擎设计，当用户从城市道路驶入山区土路或荒野区域时，由于无法匹配到道路网络，定位漂移率急剧增加，导航提示完全失效。根据测试数据，现有有路导航系统在非道路区域的定位匹配成功率低于15%，偏航检测失败率超过60%，无法提供有效的导航服务。反之，无路导航系统在道路网络完善的区域同样无法利用道路信息提升精度，导航准确性远低于有路导航系统。

第二，偏航检测与恢复机制的时效性问题。当用户偏离预设路线时，现有系统往往只能检测偏航状态，无法提供有效的偏航后导航恢复方案。以高德导航为例，当用户偏航后，系统需要等待定位稳定（约3-5秒）、重新发起路径规划请求（约2-3秒）、接收规划结果（约1-2秒）、重新计算导航引导（约0.5秒），整个偏航恢复过程耗时约7-10秒。在此期间，用户处于完全无导航状态，无法获得有效的行驶指导，用户体验较差。特别是在高速行驶场景下（时速100公里），7-10秒的无导航时间意味着车辆已行驶约200-280米，用户极易错过最佳纠偏时机。

第三，有路与无路导航切换的连续性问题。部分导航系统虽然同时支持两种导航模式，但采用切换式架构，即在某一时刻只运行一种导航引擎。当用户在有道路区域与无道路区域之间切换时，由于引擎重启和初始化需要时间，通常存在1-3秒的导航信息空白期。此外，不同导航引擎的输出格式、数据结构、状态定义各异，切换过程中可能出现导航信息中断、提示不一致、地图视图突兀变化等问题，影响导航的连续性和可靠性。

第四，多引擎协同输出的标准化问题。现有技术中，有路导航引擎和无路导航引擎通常由不同厂商开发或采用不同的技术方案，导致输出格式差异显著。有路导航引擎通常输出结构化道路信息（如道路名称、路口类型、转弯角度等），而无路导航引擎通常输出航路点信息（如航向角、航路距离等）。缺乏统一的数据封装标准，增加了上层应用开发的复杂度。例如，某主流无人机飞控系统的航路导航模块与车载导航系统之间无法直接数据共享，需要额外的格式转换模块，增加了系统复杂度和维护成本。

第五，航向角突变处理的技术缺陷。在无路导航或航路导航场景中，当航向角从359度变化到1度（或从1度变化到359度）时，简单的角度差值计算会产生约358度的错误差值，导致导航动画出现剧烈抖动。现有技术对该问题的处理通常较为简单，如直接取绝对值差或简单的阈值判断，无法准确判断是正常转向还是角度跨越，严重影响导航动画的流畅性和用户体验。

第六，导航动画的平滑性问题。现有导航系统的位置更新频率通常为1Hz（每秒1次），直接使用原始位置数据进行导航动画会导致动画卡顿、不流畅。部分系统采用简单的线性插值方法，但航向角插值存在上述360度跨越问题，导致插值结果错误。虽然tweeny等缓动库已广泛应用于UI动画领域，但在导航系统中的应用尚不充分，特别是在航向角缓动曲线的选择方面缺乏系统性的研究。

## 发明目的

本发明的目的有三个：第一，解决有路导航与无路导航融合问题，针对现有技术中单一导航引擎无法适应复杂环境的缺陷，本发明提供一种融合有路导航引擎与无路导航引擎的动态导航方法，通过双引擎并行架构实现复杂环境的无缝导航覆盖；第二，实现智能动态引擎选择机制，通过同时初始化两个导航引擎，在每次位置更新时动态比较两个引擎的匹配效果（distance2line、distance等指标），智能选择最优引擎进行导航引导，提高导航的准确性和可靠性；第三，提供平滑的偏航检测与恢复机制，当有路引擎检测到偏航时，自动切换至无路引擎继续导航，无需等待重新路径规划即可恢复导航服务，显著提升用户体验和导航连续性。

## 技术方案

本发明专利是一种融合有路导航与无路导航的动态导航方法，其核心思想是同时维护两个导航引擎（有理引擎和无路引擎），在导航过程中动态比较两个引擎的匹配效果，根据预设的优先级策略智能选择当前最优引擎进行导航输出，并在偏航等特定条件下自动切换引擎，实现无缝的导航体验。该方法的主要技术流程为双引擎初始化、并行位置更新、动态引擎选择、偏航检测与切换、统一输出封装五个阶段。

（一）双引擎并行架构
本发明采用双引擎并行架构，同时维护有路导航引擎和无路导航引擎两个独立的导航核心。有路导航引擎基于道路网络数据进行导航，主要功能包括道路网络加载与拓扑构建、GPS定位点与道路网络的匹配计算、基于已规划路径的转弯提示生成、距离路口和距离目的地等信息的计算，关键函数接口为 pie_guidance_create 创建引擎、pie_guidance_init_route 初始化路径、pie_guidance_simulate_navigation 模拟导航、pie_guidance_update_position 进行位置更新、pie_guidance_update_position_prompt 获取导航提示。无路导航引擎基于航路点数据进行导航，主要功能包括航路点数据的加载与处理、当前位置与航路线的匹配计算、航向角偏差计算、距离航路线的偏差计算，关键函数接口为 pie_guidance_noroad_create 创建引擎、pie_guidance_noroad_init_route 初始化航路、pie_guidance_noroad_simulate_navigation 模拟航路导航、pie_guidance_noroad_update_position 进行位置更新、pie_guidance_noroad_update_position_prompt 获取航路提示。引擎初始化流程包括两个阶段，首先初始化有路引擎加载道路数据并进行路径匹配初始化，其次初始化无路引擎将有路引擎的路径数据转换为节点数据后加载，转换函数为 pie_guidance_noroad_change_to_node 将路径链路转换为航路节点，两个引擎的初始化相互独立，初始化完成后统一进入导航循环。

（二）动态引擎选择算法
本发明的核心创新在于动态引擎选择算法，该算法在每次位置更新时同时调用两个引擎，并根据预设的优先级策略选择最优引擎。核心判断逻辑分为三种情况，情况一是当有路引擎返回 null 时仅使用无路引擎，若两个引擎都返回 null 则跳过本次更新继续重试，等待1秒后重试；情况二是当两个引擎都有返回时优先比较 distance2line（当前位置到规划路线的垂直距离），选择 distance2line 值较小的引擎；情况三是当 distance2line 相等时比较 distance（已行进距离），选择 distance 值较小的引擎。关键指标包括 distance2line 用于评估定位精度、distance 用于评估前进进度、route_distance 用于路线总长度用于判断是否到达终点。引擎选择决策流程为位置更新后首先判断有路引擎是否返回 null，若是有路引擎 null 则判断无路引擎是否 null，若两个都 null 则跳过本次更新，若无路引擎有返回则使用无路引擎，若两个引擎都有返回则进行 distance2line 比较，若有路引擎 distance2line 小于无路引擎则使用有路引擎否则使用无路引擎。引擎选择变量 easing_handle_use 定义为整型，取值1表示使用有路引擎、2表示使用无路引擎、3表示重新走update更新。

（三）偏航检测与自动切换机制
偏航检测与自动切换机制是本发明的重要功能，用于处理用户偏离规划路线的场景。偏航状态定义包括 PIE_GUIDANCE_STATE_DEVIATE 表示偏航状态、PIE_GUIDANCE_STATE_NORMALGUID 表示正常引导、PIE_GUIDANCE_STATE_POSTGUID 表示到达目的地后引导、PIE_GUIDANCE_STATE_PREGUID 表示预引导状态。偏航检测逻辑为当使用有路引擎时若检测到 state 等于 PIE_GUIDANCE_STATE_DEVIATE 则设置类型为偏航提示，使用无路引擎时同样检测偏航状态。自动切换触发条件包括有路引擎返回 null 表示当前位置无法匹配到道路网络、有路引擎检测到偏航 state 等于 PIE_GUIDANCE_STATE_DEVIATE、无路引擎 distance2line 更优表明当前位置更接近航路线。偏航恢复流程为有路引擎检测到偏航后提示偏航，自动切换无路引擎，基于航路点继续引导，实时计算航向角偏差，若航向角偏差超过阈值则提示偏航修正否则继续正常引导。航路导航模式下偏航检测参数包括 deviate_surface 表示水平面偏航距离阈值（单位米）、deviate_elevation 表示高程偏航距离阈值（单位米）、deviate_angle 表示角度偏航值阈值（单位度，仅在按角度导航时有效）、deviate_speed 表示有效速度阈值、arrival_dis 表示到达目的地判定距离阈值。

（四）导航动画平滑过渡
本发明设计了导航动画平滑过渡算法，用于在连续位置更新之间生成流畅的动画效果。采用基于 tweeny 缓动库的插值算法实现多维度动画插值，包括航向角缓动和距离插值两个核心维度。航向角缓动算法采用 circularOut 缓动曲线，公式为 from(eas_pre_heading).to(eas_cur_heading).via(easing::circularOut).during(25)，共25帧动画。距离插值算法采用 linear 线性缓动，公式为 from(step-SPEED).to(step).via(easing::linear).during(25)。动画帧数参数 easings_count 默认值为25帧，支持范围1-30帧可配置，每帧间隔通过 animationinterval 参数控制，默认值为40毫秒，范围为10-1001毫秒。路口距离缩放映射算法将距路口距离映射到视图缩放比例，映射区间为470米至1200米，当距离大于1200米时截断为1200米，当距离小于470米时截断为470米，映射公式为 aimscale * (MAPVIEW_PIXEL_HEIGHT / 730)，其中 MAPVIEW_PIXEL_HEIGHT 默认值为2000像素，可配置范围为大于0小于3000像素。

（五）航向角360度跨越问题的处理算法
本发明设计了航向角360度跨越处理算法，解决导航中航向角从359度到1度的突变问题。算法步骤如下：首先计算航向角差值 eas_diff_heading = eas_cur_heading - eas_pre_heading；当差值的绝对值大于180度时进行跨越处理，若差值大于0则 eas_diff_heading = eas_diff_heading - 360，否则 eas_diff_heading = 360 + eas_diff_heading；计算当前航向角 eas_cur_heading = eas_pre_heading + eas_diff_heading；最后进行边界约束处理，当 eas_cur_heading 大于360时减去360，当 eas_cur_heading 小于-360时加上360。动画帧航向角输出时同样进行边界处理，若 step_heading 小于0则加360，若 step_heading 大于360则减360，确保输出航向角在0-360度范围内。

（六）地图视图自适应缩放算法
本发明设计了地图视图自适应缩放算法，根据航向变化动态调整地图显示级别。算法核心是 elevationToMapLevel 函数，将缩放比例映射到地图级别，映射规则为 elevation 小于100对应地图级别17、elevation 小于200对应地图级别16、elevation 小于300对应地图级别15.5、elevation 小于400对应地图级别15、elevation 小于600对应地图级别14.5、elevation 小于700对应地图级别14、elevation 小于800对应地图级别13.5、elevation 小于900对应地图级别13、elevation 小于1000对应地图级别13、elevation 大于等于1000对应地图级别13。视图Y轴计算公式为 y = aimscale * (MAPVIEW_HEIGHT / 730)，加速度补偿计算为 vax_y = abs(y - last_y) * pixel_ratio，最终返回值为 y + vax_y，若返回值超过 MAPVIEW_HEIGHT 则截断为 MAPVIEW_HEIGHT。地图级别计算后乘以系数0.75和 pixel_ratio 用于最终输出。

（七）统一输出封装
为保证导航输出的统一性和一致性，本发明设计了统一的输出封装机制。无论使用哪个引擎，最终输出的导航提示信息结构统一，包括道路名称 roadname、航向角 heading、状态类型 type、距路口距离 roaddistance、距终点距离 routedistance、路口类型 crosstype、转弯类型 turntype、经度 lon、纬度 lat、速度 speed、导航状态 status 等字段，统一输出格式示例为 JSON 格式便于上层应用统一处理。输出消息类型包括 Guide_simulate_position 用于位置播报和 Guide_simulate 用于动画推送，消息版本通过 PIE_NAVIGATION_VERSION 字段标识，事件类型通过 PIE_NAVIGATION_TYPE_SIMULATE 字段标识，提示类型通过 PIE_NAVIGATION_TYPE_SIMULATE_PROMPT 标识。动画推送数据额外包含缩放比例 scale、地图级别 maplevel、历史轨迹点 easingtrace、转弯线 crossline、转弯线ID crosslineid、转弯线航向 crosslinehead 等字段。

（八）导航状态管理
本发明设计了完善的状态管理机制，协调两个引擎的工作状态。状态枚举定义包括 PIE_NAVIGATION_GUIDE_NOMAL 表示正常状态、PIE_NAVIGATION_GUIDE_PAUSE 表示暂停状态、PIE_NAVIGATION_GUIDE_START 表示启动状态、PIE_NAVIGATION_GUIDE_STOP 表示停止状态、PIE_NAVIGATION_GUIDE_CONTINUE 表示继续状态、PIE_NAVIGATION_GUIDE_NEXT_TAEGET_INDEX 表示下一目标索引。导航循环控制流程为持续检查 m_server_exit 标志判断是否退出，检查 state 判断是否为停止状态若是则退出，检查 speed 若小于1则暂停等待1秒，然后执行位置更新和引擎选择循环。提示文本通过 promptInfo->prompt_text 获取，关联到当前距目的地距离、链路层级和当前速度参数。

## 有益效果

本发明针对全地形越野导航面临的复杂环境适应性问题，提供一种融合有路导航与无路导航的动态导航方法。与现有技术相比，本发明具有以下有益效果：

本发明与现有技术的核心区别在于：在引擎架构方面，现有技术采用单一导航引擎，而本发明采用双引擎并行架构；在偏航处理方面，现有技术在偏航后需要等待重新规划路径（耗时7-10秒），本发明能够在偏航时自动切换至无路引擎继续导航（切换时间小于100毫秒）；在引擎选择方面，现有技术根据固定场景选择固定引擎，本发明通过动态比较两个引擎的 distance2line 指标智能选择最优引擎；在输出格式方面，现有技术中各引擎独立输出不同格式，本发明将不同引擎的输出统一封装为相同 JSON 格式；在状态管理方面，现有技术采用简单的状态切换，本发明采用多状态协同管理机制。

导航覆盖范围扩展：通过双引擎并行架构，实现有道路区域和无道路区域的无缝覆盖，有路引擎处理道路网络完善的区域，无路引擎处理荒野、山地、水域等无道路区域，显著扩展了导航系统的应用场景。导航准确性提升：通过动态引擎选择算法，在每次位置更新时综合比较两个引擎的匹配效果，选择 distance2line 更小的引擎进行导航输出，有效提高了定位精度和导航准确性。偏航恢复时间缩短：通过偏航自动切换机制，当用户偏离规划路线时系统自动切换至无路引擎继续导航，无需等待重新路径规划，偏航恢复时间从现有的7-10秒缩短至100毫秒以内。用户体验改善：航向角360度跨越处理算法解决了导航动画抖动问题，tweeny缓动库的应用提升了动画流畅度。系统可靠性增强：通过统一的输出封装机制，无论使用哪个引擎输出的导航信息格式一致，便于上层应用处理和维护扩展，两个引擎的并行运行提供了冗余备份提高了系统的可靠性。适配复杂应用场景：本发明特别适用于越野驾驶（野外导航无忧）、军事作战（战场环境适应性强）、无人机航路（航路点导航与地面参照双模式）、应急救援（多引擎冗余可靠性高）等复杂应用场景。

## 总结

本发明通过双引擎并行架构实现了有路导航与无路导航的动态融合，通过动态引擎选择算法智能选择最优引擎进行导航输出，通过偏航自动切换机制提供了平滑的导航恢复方案，通过航向角360度跨越处理算法解决了动画抖动问题，通过tweeny缓动库实现了导航动画的平滑过渡，通过统一输出封装保证了导航信息的格式一致性。该方法具有导航覆盖范围广、导航准确性高、偏航恢复快、动画流畅性好、系统可靠性强等特点，特别适用于全地形越野导航、航路导航、军事作战、应急救援等复杂应用场景。
