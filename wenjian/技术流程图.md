# 北斗三号特种终端导航数据编译方法 技术流程图

```mermaid
flowchart TD
    subgraph 数据源
        A[基础测绘数据] --> G
        B[GIS矢量数据] --> G
        C[POI兴趣点数据] --> G
        D[北斗定位数据] --> G
    end

    subgraph 数据预处理["步骤一：数据采集与整合"]
        G[数据整合] --> G1[格式转换<br/>SHP/KML/GeoJSON统一中间格式]
        G1 --> G2[坐标统一<br/>转换为WGS84坐标系]
        G2 --> G3[数据关联<br/>同名道路匹配融合]
        G3 --> G4[属性标准化<br/>道路等级/名称/宽度统一编码]
    end

    subgraph 数据清洗["步骤二：数据清洗与拓扑修复"]
        G4 --> H[数据清洗]
        H --> H1[剔除重复数据]
        H1 --> H2[剔除错误数据<br/>坐标越界/属性异常]
        H2 --> H3[异常值处理]
        H3 --> I[拓扑修复]
        I --> I1[断链修复<br/>弧段首尾相连]
        I1 --> I2[孤立结点处理]
        I2 --> I3[环岛/假结点处理]
        I3 --> I4[边界面处理<br/>跨网格边界连续性]
    end

    subgraph 网格划分["步骤三：网格归属计算与动态分裂"]
        I4 --> J[遍历道路弧段<br/>计算中心点一次网格]
        J --> K[统计每个一次网格<br/>道路弧段数量]
        K --> L{道路数量 ≥ 2048?}
        L -->|否| M[保持一次网格]
        L -->|是| N[8×8二次网格细分<br/>每个二次网格约47条道路]
        M --> O[网格信息写入grid_info表]
        N --> O
    end

    subgraph 拓扑构建["步骤四：拓扑关系构建"]
        O --> P[网格内拓扑]
        P --> P1[结点关联弧段<br/>填充link_ids字段]
        P1 --> P2[转向关系提取<br/>构建from/to_link连接]
        P2 --> P3[禁行规则录入<br/>单行道/限行时间]
        P3 --> Q[跨网格拓扑]
        Q --> Q1[边界点识别<br/>boundary_flag=1]
        Q1 --> Q2[邻接关系记录<br/>conn_region字段]
        Q2 --> Q3[跨网格弧段拆分<br/>分别存储对应网格]
    end

    subgraph 数据入库["步骤五：数据格式化与SQLite入库"]
        Q3 --> R[坐标正规化]
        R --> R1[绝对坐标→相对坐标<br/> Coord_norm = (Coord_abs-Coord_ref)/Coord_span]
        R1 --> S[几何压缩]
        S --> S1[差分编码+行程长度编码<br/>压缩比3:1~5:1]
        S1 --> T[数据编码]
        T --> T1[道路等级/方向等属性<br/>整数编码]
        T1 --> U[文件拆分]
        U --> U1[按网格编号生成<br/>独立SQLite文件]
        U1 --> V[SQLite建表入库]
        V --> V1[node表/link表/road表/poi表/topology表]
    end

    subgraph 索引优化["步骤六：索引创建与性能优化"]
        V1 --> W[创建空间索引]
        W --> W1[idx_grid_level/node_grid/link_grid/poi_grid]
        W1 --> X[数据库优化]
        X --> X1[ANALYZE更新统计信息]
        X1 --> X2[VACUUM整理空间]
        X2 --> X3[PRAGMA参数优化<br/>synchronous/cache_size]
    end

    subgraph 增量更新["步骤七：增量更新与维护机制"]
        X3 --> Y[版本管理<br/>每个网格维护版本号]
        Y --> Z[更新单元<br/>以网格为最小更新单元]
        Z --> AA[更新传输方式]
        AA --> AA1[北斗短报文<br/>小数据量POI变化]
        AA --> AA2[北斗数据口<br/>中等数据量更新]
        AA --> AA3[外部存储介质<br/>大数据量完整更新]
    end
```

## 专利核心技术特征对照表

| 发明点 | 流程中的体现步骤 | 技术效果 |
|--------|------------------|----------|
| **两级网格体系** | 步骤三：网格归属计算与动态分裂 | 经差1°×纬差40'一次网格，8×8二次网格细分 |
| **动态分裂规则** | 步骤三：L判断（道路数量≥2048） | 自动触发二次网格划分，数据均衡分布 |
| **正规化坐标** | 步骤五：R坐标正规化 | 精度1/65536，存储空间节省40%~60% |
| **几何压缩** | 步骤五：S几何压缩 | 差分编码+RLE，压缩比3:1~5:1 |
| **SQLite轻量存储** | 步骤五：U生成独立.db文件 | 单文件数百KB，满足特种终端资源限制 |
| **跨网格拓扑** | 步骤四：Q跨网格拓扑 | 支持跨区域路径规划 |
| **增量更新** | 步骤七：以网格为更新单元 | 更新量减少70%以上 |

## 权利要求书核心逻辑

1. **网格化数据组织** → 步骤三
2. **动态选择网格层级** → 步骤三：L判断
3. **坐标正规化压缩存储** → 步骤五：R+S
4. **SQLite轻量级数据库** → 步骤五：U+V
5. **跨网格拓扑关系** → 步骤四：Q
6. **增量更新机制** → 步骤七
