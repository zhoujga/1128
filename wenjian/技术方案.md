# 技术方案

## 1. 总体技术方案

本发明提出一种基于网格化数据组织与SQLite轻量级存储的导航数据编译方法，通过先设计逻辑组织规则，再映射为物理数据库结构，实现高效的数据访问与更新维护。该方法包括数据组织设计（逻辑模型）、数据库结构设计（物理模型）和编译流程三个主要部分。

## 2. 数据组织设计（逻辑模型）

### 2.1 网格化数据组织架构

网格化数据组织方法的核心思想是将连续的地理空间离散化划分为若干规则网格单元，每个网格单元独立存储其覆盖范围内的导航数据要素。通过网格索引，可快速定位目标区域的数据，避免全库扫描。

**具体实现方式：** 每个网格单元对应一个独立的SQLite数据库文件，网格编号作为文件名（如4265.db、426533.db），文件内部包含该网格的结点表、弧段表、道路表、POI表等数据表。采用BLOB字段存储几何数据，实现北斗三号嵌入式移动终端环境的轻量化部署。

**网格化数据组织优势：**

1. **空间索引**：通过网格编号可直接计算数据存储位置，实现O(1)时间复杂度的空间查询
2. **数据分块**：每个网格对应独立数据库文件，支持按需加载单个或多个网格数据，降低终端内存占用
3. **增量更新**：以网格为最小更新单元，仅更新变化网格的数据库文件，减少数据传输量
4. **负载均衡**：通过动态网格划分规则，保证各网格数据量均衡
5. **轻量级存储**：SQLite是北斗三号嵌入式移动终端轻量级数据库，单个数据库文件可控制在数百KB-数MB

### 2.2 层级网格划分

为平衡查询效率与数据量，采用两级网格体系：

- **一次网格**：覆盖较大区域，以经差1°、纬差40′为基本单元
- **二次网格**：在一次网格基础上细分为8×8格，每格大小为经差7.5′、纬差5′

**一次网格尺寸设计依据：**

1. 经差1°对应纬度方向约111km，满足区域级导航的覆盖需求
2. 纬差40′对应经度方向距离随纬度变化，但在中等纬度地区约对应50-60km
3. 该尺寸下单个网格的道路数据量适中，便于终端快速加载

**二次网格8×8细分设计：**

1. 每个一次网格划分为64个二次网格，保证数据细粒度管理
2. 二次网格经差7.5′、纬差5′对应实地约10×10公里，与1:2.5万地形图分幅一致
3. 便于与国家基础测绘成果对接

### 2.3 网格编号规则

**编号定义：**

- 一次网格编号采用4位整数（RRLL），RR为纬度行号，LL为经度列号，起点为纬度0°、经度60°
- 二次网格编号在一次网格编号基础上尾加两位数字（MN），M为纬度行号（0-7），N为经度行号（0-7），格式为RRLLMN

**经纬度与网格号转换公式：**

由经纬度计算网格编号：
- RR = INT(纬度 × 60 / 40)
- LL = INT(经度) - 60
- M = INT((纬度 - RR × 40/60) × 60 / 5)
- N = INT((经度 - LL - 60) × 600 / 75)

**中国区域网格规模：**

- 纬度范围：18.116° ~ 53.563°
- 经度范围：73.491° ~ 135.097°
- 一次网格规模：63列×54行，共计约3402个

### 2.4 正规化坐标

为节省存储空间并提高坐标访问效率，采用正规化坐标表示方法。

**正规化坐标计算公式：**
```
Coord_norm = (Coord_abs - Coord_ref) / Coord_span
```

**精度：**

- 一次网格：经度精度1.52587890625e-5°，纬度精度1.017252604167e-6°
- 二次网格：经度精度1.9073486328125e-6°，纬度精度1.27156575520833e-6°

### 2.5 动态网格选择规则

**规则定义：**

- 若道路数量 ＜ 2048，则该区域采用一次网格存储
- 若道路数量 ≥ 2048，则自动触发二次网格划分，将数据分散存储于64个二次网格

**阈值确定依据：**

1. 终端内存限制：特种终端可用内存通常为256MB-512MB，单个网格数据加载需控制在2-4MB以内
2. 查询性能：SQLite单表数据量超过2000条后，查询性能开始明显下降
3. 统计数据支撑：有效一次网格数量为1425个，有效二次网格数量为55469个

### 2.6 数据逻辑关系

**数据要素定义：**

- 结点（Node）：道路交叉点、端点，具有唯一编号、坐标、关联弧段列表
- 弧段（Link）：连接结点的道路片段，是导航路径计算的基本单元
- 道路元素（Road）：由若干连续弧段组成的完整道路
- 拓扑关系：定义结点与弧段、弧段与弧段之间的连接性
- 背景数据：地标、POI（兴趣点）

**要素层级关系：**
```
道路元素 → 弧段 → 结点
```

## 3. 数据库结构设计（物理模型）

### 3.1 数据类型定义

**网格编号（EPRERid）** - 联合体定义：
```c
typedef union _EPRERid {
    unsigned char mem[4];
    struct {
        unsigned RegionFlag:1;      //是否是一次网格
        unsigned ValidSign:1;      //是否为有效网格
        unsigned WRegionRow:6;     //一次网格行号
        unsigned WRegionColumn:6;  //一次网格列号
        unsigned SRegionRow:3;     //二次网格行号
        unsigned SRegionColumn:3;  //二次网格列号
        unsigned Preserve:12;      //保留
    } data;
} EPRERid;
```

**弧段编号（EPRELid）** - 结构体定义：
```c
typedef struct _EPRE_LINK_ID {
    unsigned char mem[2];
} EPRELid;
```

**结点编号（EPRENid）** - 结构体定义：
```c
typedef struct _EPRE_NODE_ID {
    unsigned char mem[2];
} EPRENid;
```

### 3.2 数据表结构

| 表名 | 说明 | 主要字段 |
|------|------|---------|
| grid_info | 网格元数据 | grid_id, level, min_lon, min_lat, road_count, node_count, link_count |
| node | 结点数据 | node_id, grid_id, lon, lat, link_ids, boundary_flag, topo_layer |
| link | 弧段数据 | link_id, grid_id, road_id, start_node, end_node, length, direction, geometry, max_speed |
| road | 道路数据 | road_id, name, link_list, road_class, width, speed_class |
| poi | 兴趣点数据 | poi_id, grid_id, lon, lat, name, type, category |
| topology | 拓扑关系 | from_link, to_link, turn_cost, turn_restrict |

**几何数据压缩存储：**

geometry字段采用差分编码+行程长度编码（RLE）的混合压缩方式：

1. 首点存储绝对坐标（经纬度）
2. 后续点存储与前一点的坐标差值
3. 对重复的差值使用行程长度编码
4. 最终数据转换为二进制BLOB存储

压缩比可达3:1~5:1，显著减少存储空间占用。

### 3.3 索引优化策略

**索引类型：**

| 索引名称 | 作用 |
|---------|------|
| idx_grid_level | 按网格级别快速筛选 |
| idx_grid_valid | 按道路数量排序查询 |
| idx_node_grid | 按所属网格定位结点 |
| idx_node_coord | 按坐标范围查询邻近结点 |
| idx_link_grid | 按所属网格定位弧段 |
| idx_link_nodes | 按起止结点查找拓扑关系 |
| idx_poi_grid | 按网格筛选POI |
| idx_poi_type | 按类型分类查询 |

## 4. 编译流程

### 4.1 数据采集与整合

**数据采集来源：**
- 北斗三号原始定位数据
- 基础测绘成果（1:5万、1:25万矢量地形图）
- GIS矢量数据（Shapefile格式）
- POI兴趣点文本数据
- 权威部门发布的道路网数据

**整合步骤：**
1. 格式转换：将SHP、KML、GeoJSON转换为统一中间格式
2. 坐标统一：转换至WGS84坐标系
3. 数据关联：匹配同一条道路的不同来源数据
4. 属性标准化：统一道路等级、名称、宽度等编码标准

### 4.2 数据清洗与拓扑修复

**数据清洗：**
1. 剔除重复数据：基于道路中心线坐标和名称进行重复检测与合并
2. 剔除错误数据：过滤坐标越界、属性异常的记录
3. 异常值处理：修正明显错误的坐标点、长度异常的道路

**拓扑修复：**
1. 断链修复：检测并修复道路弧段之间的断裂
2. 孤立结点处理：删除或关联无法构成通路的孤立结点
3. 环岛处理：识别环岛路口，创建复合结点
4. 假结点清除：合并连续且无分支的冗余结点
5. 边界面处理：确保跨网格边界道路的连续性

### 4.3 地理编码与逆地理编码处理

**地理编码（地址转坐标）：**
1. 地址分词：提取省市区、道路名、门牌号
2. 道路匹配：在道路表中进行匹配
3. 位置精确定位：结合门牌号范围计算最佳坐标

**逆地理编码（坐标转地址）：**
1. 最近道路搜索：搜索距离目标点最近的弧段
2. 匹配容差：设定距离阈值（如50米）
3. 插值计算：线性插值获取精确位置
4. POI关联：返回周边地名或兴趣点信息

### 4.4 网格归属计算与动态分裂

1. 遍历所有道路弧段，计算其中心点所属的一次网格
2. 统计每个一次网格内的道路数量
3. 若道路数量≥2048，执行二次网格分裂
4. 重新计算每条弧段所属的二次网格
5. 更新网格元数据表

### 4.5 拓扑关系构建

**网格内拓扑：**
1. 结点关联弧段：填充node表的link_ids字段
2. 转向关系提取：构建转向连接关系
3. 禁行规则录入：录入单行道、限行时间等

**跨网格拓扑：**
1. 边界点识别：标记boundary_flag=1
2. 邻接关系记录：在conn_region字段记录相邻网格编号
3. 跨网格弧段拆分：拆分为两段分别存储

### 4.6 数据格式化与入库

**格式化步骤：**
1. 坐标正规化：转换为相对网格参考点的正规化坐标
2. 几何压缩：差分编码+RLE压缩
3. 数据编码：属性编码为整数
4. 文件拆分：按网格编号生成独立.db文件

**入库操作：**
1. 创建数据库：按网格编号创建.db文件
2. 建表语句：执行CREATE TABLE
3. 数据导入：批量INSERT或COPY
4. 事务提交：每1000条提交一次

### 4.7 索引创建与性能优化

1. 运行ANALYZE语句：更新表统计信息
2. 运行VACUUM命令：回收删除空间
3. 设置PRAGMA参数：synchronous=NORMAL、cache_size等
4. 文件压缩：对.db文件进行压缩

### 4.8 更新与维护机制

**增量更新策略：**
- 更新单元：以网格为最小更新单元
- 版本管理：为每个网格维护版本号
- 更新包结构：网格编号、更新类型、变化数据、校验码

**传输方式：**
1. 北斗短报文：适用于小数据量更新
2. 北斗终端数据口：适用于中等数据量更新
3. 外部存储介质：适用于大数据量完整更新

### 4.9 测试与验证

**功能测试：**
1. 路径规划测试：验证路径计算结果正确性
2. 定位匹配测试：验证逆地理编码结果
3. 网格加载测试：验证按需加载正确性

**性能测试：**
1. 数据加载时间：目标<3秒
2. 路径规划耗时：目标<1秒
3. 内存占用：目标<100MB
